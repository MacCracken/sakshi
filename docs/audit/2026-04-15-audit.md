## CRITICAL

### SA-001: UDP msg_len written to header without clamping — remote buffer read overflow

**File**: `src/output.cyr:250-254`, `sakshi_full.cyr:262-268`
**Severity**: CRITICAL

In `_sk_write_udp_event`, `msg_len` is clamped to 256 for the `_sk_memcpy` into `pkt`, but `store16(&pkt + 10, msg_len)` writes the **original unclamped** `msg_len` into the binary header. A receiver decoding this packet would read `msg_len` from the header and attempt to read past the 256 bytes actually sent.

```
store16(&pkt + 10, msg_len);     # unclamped value in header
var pml = msg_len;
if (pml > 256) { pml = 256; }   # clamp for memcpy only
_sk_memcpy(&pkt + 12, msg, pml);
var total = 12 + pml;            # packet is 12 + clamped
```

**Impact**: Any receiver trusting the header's `msg_len` will attempt an out-of-bounds read of `msg_len - 256` bytes beyond the packet payload. In a sakshi-to-sakshi pipeline this causes memory corruption on the receiving side.

**Fix**: Write the clamped value to the header: `store16(&pkt + 10, pml)` after clamping.

**Status**: Fix in this round

## HIGH

### SA-002: `_sk_fmt_int` buffer requires 24 bytes — callers pass exactly 24 but document none

**File**: `src/format.cyr:55-69`, `sakshi.cyr:75-87`, `sakshi_full.cyr:111-123`
**Severity**: HIGH

`_sk_fmt_int(n, buf)` writes into `buf` at offsets 0..23. The function's callers (`_sk_write_int`) declare `var tmp[24]` which is correct. However, `_sk_fmt_int` has no bounds check — it trusts `buf` is at least 24 bytes. If any future caller passes a smaller buffer, silent memory corruption occurs. The function comment says "Buffer must be at least 24 bytes" in `format.cyr` but the distribution profiles have no such comment.

**Impact**: No current exploit — all existing callers are correct. Future misuse would cause stack corruption.

**Fix**: Add comment to distribution profiles. Consider defensive: `if (bi < 0) { return 0; }` guard in the write loop (LOW priority, deferred).

**Status**: Deferred — comment fix in this round

### SA-003: `sakshi_ring_read_raw` trusts caller-provided `dst` pointer and `max_len`

**File**: `src/output.cyr:126-137`, `sakshi_full.cyr:427-438`
**Severity**: HIGH

`sakshi_ring_read_raw(dst, max_len)` performs `store8(dst + ri, ...)` in a loop up to `max_len` bytes. If `dst` is null or points to an undersized buffer, this writes out of bounds. This is a public API function — consumers call it directly.

**Impact**: Caller error causes arbitrary memory write. Since this is a library API (not internal), consumers can misuse it.

**Fix**: This matches C library convention (caller responsible for buffer size). Document the contract clearly. No runtime fix needed — validation at system boundary is the consumer's responsibility.

**Status**: Documentation fix in this round

### SA-004: `sakshi_ring_decode_event` reads `msg_len` from untrusted ring buffer data — no upper bound check

**File**: `src/output.cyr:160-198`, `sakshi_full.cyr:448-471`
**Severity**: HIGH

`sakshi_ring_decode_event` reads `msg_len = load16(buf + 10)` from the binary event header in the ring buffer. It checks `buf_len < event_size` (where `event_size = 12 + msg_len`), which prevents reading past the input buffer. However, `msg_len` is a u16 (max 65535), and the output write loop `_sk_write_str(dst, pos, dst_cap, buf + _SK_BIN_HDR, msg_len)` is bounded by `dst_cap`, so the output side is safe. The **input** side reads `buf + 12` through `buf + 12 + msg_len`, which is bounded by the `buf_len < event_size` check.

On closer analysis: both input and output paths are bounded. The ring buffer itself can only contain events up to 4096 bytes total, so `msg_len` is naturally bounded by ring geometry. However, if the ring buffer has been corrupted (wrapped mid-event, overwrite-oldest), the header fields may be garbage. The `buf_len < event_size` check catches this for overflow (event_size wraps or exceeds remaining data), but a corrupted `msg_len` that is small enough to pass the check would cause the decoder to consume the wrong number of bytes, misaligning all subsequent event reads.

**Impact**: Ring buffer corruption after overwrite can cause misaligned decode chain. Not exploitable for arbitrary code execution but causes data integrity loss in decoded output.

**Fix**: Add a sanity check: `if (msg_len > 4084) { return 0; }` (max message that fits in ring = 4096 - 12 header). Deferred to next round.

**Status**: Deferred — roadmap item

## MEDIUM

### SA-005: `_sk_now_ns` timestamp multiplication can overflow i64

**File**: `src/format.cyr:44-47`, `sakshi.cyr:69-73`, `sakshi_full.cyr:105-109`
**Severity**: MEDIUM

`load64(&ts) * 1000000000 + load64(&ts + 8)` — the seconds field from `clock_gettime` multiplied by 1 billion. An i64 overflows at ~9.2 * 10^18 nanoseconds = ~292 years from epoch. Since this uses `CLOCK_MONOTONIC_RAW` (time since boot), overflow requires ~292 years of uptime. Not exploitable in practice, but if used on a system with a broken monotonic clock, the timestamp could wrap to negative and affect log ordering.

**Impact**: Theoretical only. Would require 292 years of uptime or broken clock hardware.

**Fix**: No fix needed. Document the 292-year limit as a known constraint.

**Status**: Deferred — documentation only

### SA-006: `sakshi_output_file` does not validate path — potential path traversal

**File**: `src/output.cyr:35-41`, `sakshi.cyr:155-161`, `sakshi_full.cyr:408-414`
**Severity**: MEDIUM

`sakshi_output_file(path)` passes `path` directly to `syscall(2, path, ...)` (open). No validation of the path contents. If a consumer passes user-controlled input as the path, `../` traversal or writing to arbitrary locations is possible.

**Impact**: Sakshi is a library — the caller controls the path. This is the same trust boundary as C's `fopen()`. The risk exists only if a consumer passes unsanitized user input as the log file path.

**Fix**: Document that `path` must be a trusted, hardcoded or config-derived path. Path validation is the consumer's responsibility. No runtime fix in the library.

**Status**: Documentation fix in this round

### SA-007: `sakshi_set_output_fd` accepts any integer — no fd validation

**File**: `src/output.cyr:338-341`, `sakshi.cyr:152`, `sakshi_full.cyr:401-405`
**Severity**: MEDIUM

`sakshi_set_output_fd(fd)` stores the fd directly into `_sk_stderr_fd` with no validation. A negative fd or an fd pointing to a closed file will cause write syscalls to silently fail (write returns -EBADF), which is harmless but means log data is silently lost.

**Impact**: Silent log loss if caller passes invalid fd. No memory corruption or security exploit.

**Fix**: Document the contract. Optionally validate `fd >= 0` at entry. Deferred.

**Status**: Deferred — LOW priority

### SA-008: Slim profile `_sk_emit` differs from modular — level check in wrong place

**File**: `sakshi.cyr:174-183`
**Severity**: MEDIUM

In the slim profile, `_sk_emit` performs the level check (`if (level > _sk_log_level) { return 0; }`), while in the modular source, the level check is in `_sk_log` (trace.cyr:38) and `_sk_emit` is level-agnostic. This means in the slim profile, the level check happens inside `_sk_emit`, while in the full profile it happens in `_sk_log` before `_sk_emit` is called. The behavior is functionally equivalent for the trace API, but `_sk_emit` in the slim profile will silently filter span events if someone calls it directly with a level > log_level. In the modular/full profile, `_sk_emit_span` bypasses level filtering (spans are always emitted regardless of log level), which is the correct behavior.

This is not currently exploitable because the slim profile has no span API, but it represents a behavioral divergence that could cause confusion if spans are ever added to the slim profile.

**Impact**: No current impact. Architectural inconsistency between profiles.

**Fix**: No fix needed for current scope. Note for future slim profile expansion.

**Status**: Deferred — roadmap note

## LOW

### SA-009: `_sk_strlen` has no upper bound — potential infinite loop on non-null-terminated input

**File**: `src/format.cyr:10-13`, `sakshi_full.cyr:94-98`
**Severity**: LOW

`_sk_strlen(s)` scans until it finds a null byte. If `s` points to non-null-terminated memory, this loops until it hits a null byte or segfaults. The function is internal (`_sk_`) and currently unused by any code path (DCE reports it as dead code), but it remains in the source.

**Impact**: None — function is dead code. Would only matter if called with untrusted input.

**Fix**: Remove dead code or add a max_len parameter if revived. Deferred.

**Status**: Deferred — dead code removal candidate

### SA-010: `clock_gettime` return value unchecked

**File**: `src/format.cyr:45-46`, `sakshi.cyr:70-72`, `sakshi_full.cyr:106-108`
**Severity**: LOW

`syscall(228, 4, &ts)` — the return value of clock_gettime is not checked. If the syscall fails (returns negative), `ts` contains uninitialized stack data, and the computed timestamp is garbage.

**Impact**: `clock_gettime(CLOCK_MONOTONIC_RAW)` effectively never fails on Linux (kernel guarantees the clock exists). If it did fail, the only consequence is garbage timestamps — no memory corruption.

**Fix**: Optionally check return value and use 0 as fallback timestamp. Deferred — extremely low probability.

**Status**: Deferred

### SA-011: `_sk_ring_write` counter grows unbounded — u64 wrap after ~18 quintillion writes

**File**: `src/output.cyr:67,81`, `sakshi_full.cyr:83,229`
**Severity**: LOW

`_sk_ring_write` is incremented on every byte written to the ring buffer and never wraps explicitly. It relies on natural i64 overflow. At 1 million events/second with 20-byte events, this wraps in ~29,000 years. The ring buffer indexing uses `& _SK_RING_MASK` which is correct regardless of the counter magnitude.

**Impact**: No practical impact. The mask-based indexing is overflow-safe by design.

**Fix**: None needed. The design is correct.

**Status**: No action

## CVE/0day Pattern Review

### Patterns checked against known CVEs in logging/tracing libraries:

| Pattern | CVE Examples | Sakshi Status |
|---------|-------------|---------------|
| **Log injection** (CVE-2021-44228/Log4Shell, CVE-2021-45046) | Log4j JNDI lookup injection | **Not vulnerable** — sakshi has no string interpolation, no lookup mechanisms, no format string parsing. Messages are raw byte arrays written verbatim. |
| **Format string attacks** (CVE-2000-0573, CWE-134) | printf-style format string injection | **Not vulnerable** — no format string parsing. All formatting is internal integer-to-decimal conversion. Messages are opaque byte blobs. |
| **Ring buffer corruption / race conditions** (CVE-2020-14386 kernel ring buffer) | Concurrent access to shared ring buffer | **Not vulnerable in current design** — sakshi is single-threaded. No locks needed. If used in multi-threaded context, ring buffer would need synchronization. Document as single-threaded-only. |
| **UDP amplification / reflection** (CVE-2013-5211 NTP monlist) | Attacker spoofs source IP to amplify traffic | **Low risk** — sakshi only sends log events, never responds to incoming packets. No amplification vector. However, log data sent over UDP is unencrypted and unauthenticated. |
| **Integer overflow in size fields** (CVE-2021-3156 sudoedit heap overflow) | Crafted size values cause buffer overflow | **SA-001 applies** — `msg_len` in UDP header is unclamped. SA-004 addresses ring buffer decode. |
| **Symlink/TOCTOU on file output** (CVE-2008-0166, CWE-367) | Attacker replaces log file with symlink between check and write | **Low risk** — sakshi opens file once with O_APPEND. No TOCTOU window on the open itself, but a pre-existing symlink at the path would be followed. This is standard POSIX behavior and matches C library convention. |
| **Uninitialized memory disclosure** (CVE-2017-1000364) | Stack buffer sent to network contains prior stack data | **SA-001 related** — in `_sk_write_udp_event`, `var pkt[268]` is not zeroed. If `msg_len` < 256, only `12 + msg_len` bytes are sent, which is correct. No uninitialized bytes are transmitted. **Clean.** |
